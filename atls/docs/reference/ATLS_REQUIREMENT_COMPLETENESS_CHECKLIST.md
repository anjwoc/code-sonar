# ATLS Requirement Completeness Checklist

## 목적

이 체크리스트는 `Jira 본문`, `Jira 코멘트`, `등록된 source 문서`를 종합해 requirement를 만들 때,
**열거된 조건과 단계가 누락되지 않았는지** 확인하기 위한 최소 검토 규칙이다.

특히 QA 이슈처럼 코멘트 안에 실제 구현/검증 조건이 정리되어 있는 경우,
문장을 부분 요약하는 대신 **열거된 acceptance 조건 전체를 case로 확장했는지** 확인해야 한다.

## 적용 대상

- Jira 코멘트에 번호 목록이나 bullet로 기대 동작이 정리된 이슈
- 본문보다 코멘트가 더 구체적으로 조건을 설명하는 이슈
- “초기 상태 / 중간 상태 / 마지막 상태”처럼 다단계 UI 흐름이 있는 이슈
- 숫자 옵션, 페이지/리스트 이동, 화살표 노출 조건처럼 조건 분기가 중요한 이슈

## 필수 체크

### 1. Enumerated Condition Completeness

- 코멘트에 `1. 2. 3.` 또는 `-` `*` `•` 형태로 정리된 항목이 있으면, 각 항목을 독립 조건으로 추출했는가
- 하나의 문단 안에 여러 조건이 섞여 있으면, 조건별로 분리했는가
- `최초`, `클릭 후`, `이동 후`, `마지막`, `n번째` 같은 상태 변화 단서를 별도 조건으로 보존했는가

### 2. Acceptance Expansion

- 추출한 조건이 acceptance case의 `expected` 또는 별도 case로 실제 내려갔는가
- “처음 상태”, “중간 상태”, “끝 상태”가 모두 검증 범위에 포함됐는가
- `forbidden`에도 대표 오동작이 적혀 있는가

### 3. Non-Requirement Filtering

- 인사말, 감사 문구, 확인 요청, 단순 참고 링크는 requirement에서 제외했는가
- `사전 조건`, `재현 경로`, `계정 정보`는 테스트 실행 정보이지 제품 요구사항이 아님을 구분했는가

### 4. Completion Gate

다음 중 하나라도 `아니오`면 해당 이슈는 “완료”로 보면 안 된다.

- 코멘트의 열거 조건이 빠짐없이 requirement로 정리되었는가
- requirement가 acceptance case로 실제 확장되었는가
- acceptance case를 기준으로 E2E와 evidence(video/screenshot)가 남았는가
- evidence가 해당 조건을 실제로 검증했는가

## QA-10719 같은 사례에서의 적용 예

코멘트:

- 최초 노출시에는 화살표 없음
- 리스트 당 칩 5개 초과 시 우측 화살표 노출
- 우측화살표 클릭 시 첫 리스트 짤린 칩이 맨 앞에 노출
- 2번째 리스트에서 3번째 리스트까지 갈 수 있으면 좌우 화살표 노출
- 마지막 리스트에서 칩이 5개 초과하지 않으면 왼쪽 화살표만 노출

이 경우 requirement/acceptance는 최소한 위 5개를 모두 포함해야 한다.

다음은 불완전한 상태다.

- “overflow면 우측 화살표 노출”까지만 구현/검증
- `최초 없음`, `중간 리스트 좌우`, `마지막 리스트 왼쪽만`을 빠뜨림
- 코멘트를 읽었지만 일부만 구현하고 완료로 판단

## 운영 규칙

- 코멘트에 열거 조건이 있으면, 해당 이슈는 기본적으로 `partial`로 시작한다.
- 열거 조건 전체가 acceptance case로 확장되기 전까지 `full` 또는 `done`으로 올리지 않는다.
- evidence가 남더라도, 그 evidence가 일부 조건만 검증했다면 “부분 검증”으로 표시한다.
